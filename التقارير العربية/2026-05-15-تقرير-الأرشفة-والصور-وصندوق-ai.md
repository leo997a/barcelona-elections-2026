# تقرير 2026-05-15: الأرشفة، صور اللاعبين، وصندوق الذكاء الاصطناعي

## 1. التوضيح المهم حول المفاتيح

`REO_ARCHIVE_GITHUB_TOKEN` ليس مفتاح ذكاء اصطناعي. هذا مفتاح GitHub موجود داخل Google Cloud VM ويعطي جسر استخراج WhoScored صلاحية كتابة ملفات JSON داخل مستودع GitHub. لا يمكن "جلب" قيمة هذا المفتاح مرة أخرى بعد إنشائه من GitHub؛ إذا ضاع يتم إنشاء Fine-grained token جديد بصلاحية `Contents: Read and write` على المستودع المطلوب فقط.

`SPORTMONKS_API_TOKEN` مفتاح مختلف تماما. هو لمصدر بيانات خارجي اختياري وليس مستخدما في تشغيل مباراة WhoScored الحالية. تركت مصدر البيانات كما طلبت: WhoScored عبر الجسر والاستخراج الذكي فقط.

مفاتيح Gemini أو Google AI يجب ألا توضع في الكود أو في المتصفح. إذا أردنا استخدام عدة مفاتيح، المكان الصحيح هو متغيرات البيئة على Vercel أو خلف LiteLLM Proxy، مثلا:

```env
GEMINI_API_KEYS=key1,key2,key3
LITELLM_PROXY_URL=https://your-litellm-proxy.example.com
LITELLM_API_KEY=server-side-proxy-key
LITELLM_MODELS=gemini/gemini-2.5-flash,gemini/gemini-2.5-flash-lite
```

أي مفاتيح تم إرسالها في محادثة أو ظهرت على الشاشة الأفضل تدويرها لاحقا من Google AI Studio، ثم إضافتها كمتغير بيئة فقط.

## 2. أين يتم تخزين المباراة الآن؟

التدفق بعد التعديل أصبح كالتالي:

1. القالب أو لوحة التحكم يرسل رابط WhoScored إلى Vercel.
2. Vercel يمرر الأمر إلى Google Cloud Bridge.
3. Google Cloud Bridge يشغل extractor ويحفظ آخر لقطة داخل VM في ملف محلي:
   `data/latest-match.json`
4. بعد كل استخراج ناجح، إذا تغيرت البيانات، الجسر يرفع نسخة JSON إلى GitHub داخل:
   `match-archive/<season>/<competition>/<round>/<date>_<matchId>_<home>-vs-<away>.json`
5. عند الضغط على زر "أرشفة الآن في GitHub" يتم إجبار الجسر على حفظ آخر لقطة حتى لو كان الاستخراج متوقفا.

بهذا لا تبقى المباراة محصورة في ذاكرة السيرفر أو في جهازك. كل مباراة ناجحة تصبح قابلة للرجوع من GitHub لاحقا، ويمكن بناء مكتبة موسمية منظمة عليها.

ملاحظة تشغيلية: إذا رفض GitHub التوكن بسبب صلاحيات أو انتهائه، الجسر لا يرمي البيانات؛ يحفظ نسخة انتظار داخل الـ VM في:

```text
data/archive-queue/<archive-path>.json
```

ثم يظهر `queuedPath` في حالة الأرشيف حتى نبدل التوكن بتوكن GitHub صحيح.

## 3. لماذا لم تكن الأرشفة ظاهرة؟

الكود السابق كان يؤرشف غالبا عند نهاية المباراة فقط. إذا كانت المباراة انتهت، أو العامل توقف، أو لم يمر بدورة "final" بالطريقة المتوقعة، يبقى `archive` فارغا. التعديل الجديد نقل الأرشفة من "بعد صافرة النهاية فقط" إلى "بعد كل استخراج ناجح ومتغير"، مع بصمة للبيانات تمنع حفظ نفس اللقطة بلا فائدة.

أضفت أيضا endpoint جديد:

```text
POST /api/control/archive
```

ومررته عبر Vercel:

```text
POST /api/reo-match/control?action=archive
```

## 4. صور اللاعبين والكاش

سبب عدم ظهور الصور في القوالب القديمة أن بعض القوالب لا تحمل حقل `playerImageCacheUrl` أصلا، وبالتالي القالب لم يكن يجلب ملف الكاش حتى لو كان موجودا. تم جعل القالب يستخدم هذا المسار افتراضيا:

```text
/player-image-cache/barcelona.json?v=20260515
```

أي قالب Match Stats قديم سيحاول الآن تحميل كاش برشلونة تلقائيا. تم تحسين المطابقة كذلك لتجرب:

- رابط صورة مباشر إن وجد.
- `player.id` و `playerId`.
- الاسم الكامل.
- اسم العائلة.
- صيغة الحرف الأول + اسم العائلة مثل `J Kounde` و `J. Kounde`.

الفحص الحالي لمجلد برشلونة في مستودع الصور وجد صورا مثل: Koundé, Casadó, Bernal, Olmo, Balde, Pedri, Lewandowski. اللاعبون غير الموجودين في مجلد الصور سيبقون باختصار الاسم إلى أن نضيف صورهم للكاش.

## 5. تجهيز صندوق AI للقوالب الحالية والمستقبلية

تم تطوير endpoint الذكاء الاصطناعي `/api/ai` ليعمل كطبقة خلفية مرنة:

- يدعم مفتاح Gemini واحد عبر `GEMINI_API_KEY`.
- يدعم عدة مفاتيح عبر `GEMINI_API_KEYS` مع fallback تلقائي.
- يدعم LiteLLM Proxy عبر `LITELLM_PROXY_URL` و `LITELLM_MODELS`.
- يضع timeout للطلبات حتى لا تتجمد الواجهة.
- يظل كل شيء server-side ولا يظهر في bundle المتصفح.

أضيفت بدايتان مهمتان للقوالب:

### `template-assist`

لتحويل نص أو صورة إلى حقول جاهزة لأي قالب بث:

- عنوان.
- سطر داعم.
- وسوم.
- حقول قابلة للحقن داخل القالب.
- ملاحظات تحريرية.

### `player-transfer-card`

لبناء بطاقة ميركاتو من اسم لاعب أو نص خبر:

- اسم اللاعب.
- النادي.
- المركز.
- عنوان ميركاتو.
- ملخص.
- إحصائيات إذا كانت موجودة في النص.
- تلميحات بحث لصورة اللاعب والكاش.

هذه لا تغير مصدر بيانات WhoScored الحالي. هي أساس هندسي كي ندخل لاحقا أدوات مثل: قراءة صورة خبر، استخراج العنوان، تحويله لقالب، أو البحث عن لاعب ثم بناء بطاقة انتقالات.

## 6. خطة صور اللاعبين القادمة

أفضل نظام عملي:

1. نعتبر GitHub repo `graphicsplayer2026` هو مصدر كاش الصور الذي نثق به.
2. ننشئ ملف JSON لكل فريق:
   `public/player-image-cache/barcelona.json`
3. كل عنصر فيه يربط أسماء اللاعب وصيغها المختلفة برابط raw مباشر.
4. عند إدخال لاعب جديد، AI يقترح مفاتيح المطابقة، لكن لا يعتمد عليه وحده.
5. إن استخدمنا مصادر خارجية مثل SportRenders، الأفضل أن نحول الصورة إلى كاش منظّم عندنا بدل الاعتماد على scraping مباشر أثناء البث.

بهذا القالب لا يتقطع ولا ينتظر مواقع خارجية كثيرة أثناء العرض، وكل صورة تصبح ثابتة وسريعة.

## 7. ما الذي يجب عمله لاحقا لتطوير القالب أكثر؟

اقتراحات 2026 العملية:

- نمط "Transfer Reveal": بطاقة لاعب تظهر ككشف تلفزيوني مع صورة كبيرة، شعار النادي، ومؤشر ثقة الخبر.
- نمط "Data Duel": مقارنة لاعب مع بديل محتمل في نفس المركز.
- نمط "Rumor Timeline": خط زمني للخبر من مصدره الأول حتى آخر تحديث.
- نمط "Player Snapshot": صورة لاعب + 5 أرقام موسمية + نقاط قوة بدون تقييم رقمي عام.
- نمط "Breaking Lower Third": شريط سفلي سريع للأخبار العاجلة.
- نمط "Tactical Fit": أين يلعب اللاعب في تشكيلة الفريق مع 3 أسباب فنية.

الأهم أن تكون كل هذه القوالب مرتبطة بصندوق AI واحد: يدخل النص أو الصورة أو اسم اللاعب، ثم يخرج حقولا منظمة، والقالب فقط يعرضها بسلاسة.
