# تقرير عربي شامل - أرشفة المباريات وكاش الصور وخطة تطوير القوالب

التاريخ: 2026-05-14  
المشروع: `barcelona-elections-2026`  
النطاق: جسر استخراج بيانات المباريات، أرشفة GitHub، كاش صور اللاعبين، وتحسين قالب إحصائيات المباراة المباشر.

## 1. ماذا تم بناؤه فعليا؟

تم تحويل قالب إحصائيات المباراة من قالب يعتمد على إدخال JSON يدوي إلى نظام بث مباشر فيه ثلاث طبقات:

1. طبقة الاستخراج:
   - أداة Python تقرأ بيانات المباراة من WhoScored عندما يكون رابط المباراة صحيحا.
   - تستخرج النتيجة، الدقيقة، حالة المباراة، أحداث المباراة، إحصائيات الفرق، إحصائيات اللاعبين، الشعارات، وبعض المؤشرات المركبة.

2. طبقة الجسر السحابي:
   - خدمة `reo-match-bridge` تعمل على Google Cloud.
   - تحفظ الرابط الحالي وبيانات آخر مباراة.
   - يستطيع موقع Vercel الاتصال بها عبر API بدون تشغيل الأداة محليا.

3. طبقة القالب في الموقع:
   - قالب Match Stats يقرأ من Cloud Bridge.
   - يعرض لوحة المباراة، لوحة اللاعبين، أحداث المباراة، الأفضلية، الإحصائيات المتغيرة، والشعارات.
   - تم تثبيت التحديث حتى لا يظهر كأنه يعمل refresh أو تقطيع كل بضع ثوان.

## 2. أين تخزن بيانات المباراة الآن؟

التخزين الحالي ليس داخل Vercel. Vercel هو واجهة عرض ووسيط API فقط.

الأماكن الفعلية:

- داخل Google Cloud:
  `/opt/reo-match-bridge/data/latest-match.json`

- رابط المباراة الحالي داخل Google Cloud:
  `/opt/reo-match-bridge/data/current-match.json`

- عند التشغيل المحلي للأداة:
  `extractor/output/latest_match_stats.json`

- عند الاستخراج اليدوي المحلي:
  `extractor/output/match_<team>_<timestamp>.json`

الموقع يقرأ من:

`/api/reo-match/match`

وهذا المسار في Vercel يذهب إلى جسر Google Cloud، ثم يرجع آخر بيانات مباراة.

## 3. هل يتم تخزين المباريات في GitHub؟

الكود أصبح جاهزا للأرشفة في GitHub، لكنه لا يعمل إلا بعد وضع متغيرات سرية على سيرفر Google Cloud.

المتغيرات المطلوبة:

```txt
REO_ARCHIVE_GITHUB_TOKEN=ضع_التوكن_هنا
REO_ARCHIVE_GITHUB_REPO=leo997a/barcelona-elections-2026
REO_ARCHIVE_GITHUB_BRANCH=main
REO_ARCHIVE_BASE_PATH=match-archive
```

بعد التفعيل، عند نهاية المباراة يحفظ الجسر نسخة JSON داخل GitHub بمسار منظم تقريبا بهذا الشكل:

```txt
match-archive/2025-2026/spain-laliga/round-unknown/2026-05-14_1914217_deportivo-alaves-vs-barcelona.json
```

إذا كان الملف موجودا مسبقا، الجسر يحدثه بدلا من أن يفشل بسبب وجود ملف بنفس الاسم.

## 4. كيف تحصل على `REO_ARCHIVE_GITHUB_TOKEN`؟

الأفضل استخدام Fine-grained Personal Access Token لأنه يمكن حصره على مستودع واحد وصلاحية واحدة.

الخطوات:

1. افتح GitHub.
2. اضغط على صورتك الشخصية في الأعلى.
3. ادخل إلى `Settings`.
4. ادخل إلى `Developer settings`.
5. افتح `Personal access tokens`.
6. اختر `Fine-grained tokens`.
7. اضغط `Generate new token`.
8. الاسم المقترح:
   `reo-match-archive`
9. اختر مدة صلاحية مناسبة، مثل 90 أو 180 يوم.
10. في `Repository access` اختر:
    `Only select repositories`
11. اختر هذا المستودع فقط:
    `leo997a/barcelona-elections-2026`
12. في `Repository permissions` اجعل:
    `Contents = Read and write`
13. اترك `Metadata` كما هو، غالبا يكون Read فقط.
14. اضغط Generate.
15. انسخ التوكن مرة واحدة فقط، لأن GitHub لن يعرضه مرة ثانية.

مهم جدا:

- لا تضع التوكن في كود الموقع.
- لا تضعه في Vercel.
- لا تكتبه داخل JSON أو داخل إعدادات القالب.
- مكانه الصحيح هو ملف البيئة الخاص بخدمة Google Cloud فقط.

## 5. كيف تفعّل الأرشفة على Google Cloud؟

من جهازك:

```powershell
gcloud compute ssh openclaw-server --zone us-west1-a
```

داخل السيرفر:

```bash
sudo nano /etc/reo-match-bridge.env
```

أضف:

```bash
REO_ARCHIVE_GITHUB_TOKEN=التوكن_الذي_نسخته
REO_ARCHIVE_GITHUB_REPO=leo997a/barcelona-elections-2026
REO_ARCHIVE_GITHUB_BRANCH=main
REO_ARCHIVE_BASE_PATH=match-archive
```

ثم:

```bash
sudo systemctl restart reo-match-bridge
sudo systemctl status reo-match-bridge --no-pager
```

للفحص:

```bash
curl http://127.0.0.1:3005/api/status
```

عند نهاية مباراة، يفترض أن ترى قيمة `archive` في الحالة، أو يظهر ملف جديد داخل GitHub تحت مجلد `match-archive`.

## 6. ماذا تفعل في صفحة Branches الظاهرة في الصورة؟

الصورة تعني:

- الفرع الرئيسي هو `main`.
- يوجد فرع تطوير باسم:
  `codex/broadcast-template-upgrades`
- هذا الفرع أمام `main` بعدة commits.
- يوجد Pull Request رقم `#1`.

الإجراء الصحيح:

1. افتح Pull Request رقم `#1`.
2. تأكد أن الفحوصات خضراء.
3. اضغط Merge إذا أردت أن تصبح التغييرات جزءا من `main`.
4. خيار `Protect this branch` مفيد لاحقا، لكنه ليس ضروريا الآن للأداة. يمكن تفعيله عندما تريد منع الدفع المباشر على `main` وإجبار كل شيء أن يمر عبر Pull Request.

## 7. نظام كاش صور اللاعبين

تم ربط القالب بملف كاش افتراضي:

```txt
public/player-image-cache/barcelona.json
```

مصدر الصور الحالي:

```txt
https://github.com/leo997a/graphicsplayer2026/tree/main/La%20Liga/Barcelona
```

تمت إضافة أمر مزامنة جديد:

```bash
node scripts/sync-player-image-cache.mjs --path="La Liga/Barcelona" --out="public/player-image-cache/barcelona.json"
```

هذا الأمر:

- يقرأ ملفات الصور من مستودع `graphicsplayer2026`.
- يستخدم `Barcelona.json` إذا كان موجودا.
- يضيف أسماء بديلة بدون علامات مثل:
  `Kounde` و `Koundé`
  `Cubarsi` و `Cubarsí`
  `Casado` و `Casadó`
- يكتب ملف كاش مرتب داخل المشروع.

تم أيضا تحسين مطابقة أسماء اللاعبين داخل القالب حتى يتعامل مع اختلاف العلامات بين مصدر الصور وWhoScored.

## 8. لماذا لا نقرأ الصور مباشرة من الموقع أثناء البث؟

لأن البث يحتاج ثباتا، وليس مجرد أن تعمل الصورة مرة واحدة.

المشكلة في الاعتماد المباشر على مواقع خارجية:

- قد تمنع hotlink.
- قد تكون بطيئة وقت البث.
- قد تغير أسماء الملفات.
- قد ترجع 403 أو 404 فجأة.
- قد تؤثر على سلاسة القالب.

الحل الأفضل:

1. نخزن الصور في مستودع نملكه أو CDN نملكه.
2. ننشئ JSON لكل فريق.
3. القالب يقرأ JSON مرة واحدة بكاش المتصفح.
4. إن لم يجد صورة، يستخدم fallback آمن بدون كسر القالب.

لاحقا يمكن جعل البنية:

```txt
public/player-image-cache/
  la-liga/
    barcelona.json
    real-madrid.json
    atletico-madrid.json
  premier-league/
    man-city.json
    arsenal.json
```

## 9. كيف نستفيد من أرشفة المباريات لاحقا؟

عندما تبدأ المباريات بالتخزين في GitHub، لن تكون مجرد ملفات. ستصبح قاعدة بيانات موسمية.

أمثلة استخدام:

- أرشيف موسم كامل لكل فريق.
- مقارنة أداء برشلونة من مباراة إلى مباراة.
- استخراج أفضل لاعبي الموسم حسب التسديدات، التمريرات المفتاحية، الاسترجاعات، العرضيات، الاعتراضات.
- بناء تقارير بعد المباراة تلقائيا.
- إنشاء قوالب قبل المباراة بناء على آخر 5 مباريات.
- بناء إحصائيات لاعب ضد فريق معين.
- استخراج "قصة المباراة" تلقائيا: من سيطر؟ متى تغير الزخم؟ من صنع الفارق؟
- تغذية قوالب جديدة مثل:
  Player Form
  Team Trend
  Season Dashboard
  Rival Watch
  Match Archive Replay

الأرشيف يمكن أن يخدم أيضا:

- برامج تحليل أسبوعية.
- فيديوهات قصيرة.
- نشرات قبل المباراة.
- مقارنة المدربين.
- متابعة اللاعبين الشباب.
- ملخصات انتقالات مبنية على أداء فعلي.

## 10. هل الموضوع خاص بالمباريات فقط؟

لا. نفس البنية يمكن توسيعها لبيانات أخرى:

- مباريات مباشرة.
- مباريات مؤرشفة.
- قائمة مباريات الموسم.
- ترتيب الدوري.
- بيانات الفرق.
- بيانات اللاعبين.
- صور اللاعبين.
- شعارات الأندية.
- تقارير قبل المباراة.
- بيانات انتقالات.
- أخبار السوق.

الفكرة الهندسية هي بناء "مخزن بيانات بث" وليس أداة مباراة واحدة فقط.

## 11. لماذا حدث التقطيع سابقا؟

السبب الأقرب كان في طريقة تحديث نافذة العرض:

- كانت تعمل polling سريع.
- أحيانا تعيد تطبيق نفس حالة القالب بدون تغير فعلي.
- الأنميشن كان يلمع ويتحرك بسرعة، فيبدو للمشاهد أن القالب يعمل refresh.
- بعض اللوحات كانت قابلة للضغط العمودي، فتظهر مقطوعة في دقة العرض.

تمت المعالجة عبر:

- منع تطبيق نفس البيانات مرتين إذا لم تتغير.
- إبطاء fallback polling.
- إزالة حلقة تحديث مكررة.
- جعل اللوحات المهمة `shrink-0`.
- تثبيت ارتفاعات داخلية.
- إبطاء اللمعان والحركة.

## 12. كيف يمكن تطوير القالب أكثر؟

أفكار عملية للمرحلة القادمة:

1. Story Engine
   - القالب لا يعرض أرقاما فقط، بل يقرر القصة:
     "الفريق متقدم لأنه أكثر خطورة في الثلث الأخير"
     "أفضل لاعب في الضغط"
     "أكثر لاعب يصنع فرصا"

2. Tactical Cards
   - بطاقات صغيرة تشرح:
     منطقة السيطرة
     اللاعب المفتاحي
     نقطة ضعف الفريق
     هل المباراة مفتوحة أم مغلقة

3. Player Spotlight
   - عندما يظهر لاعب في قائمة الأفضل، يظهر وجهه، رقمه، فريقه، وأيقونة الإحصائية.

4. Momentum Timeline
   - خط زمني يعرض فترات السيطرة أو الأحداث المهمة.

5. Smart Rotation
   - ليس دوران عشوائي كل 30 ثانية.
   - إذا حدث هدف أو بطاقة، يوقف الدوران مؤقتا ويعرض الحدث.
   - إذا انتهى الشوط، ينتقل إلى لوحة ملخص الشوط.

6. Presenter Safe Zone
   - منطقة وسطية مخصصة لصانع المحتوى حتى لا تغطيه اللوحات.
   - القالب يلتف حول هذه المنطقة تلقائيا.

7. Multi Theme System
   - نماذج بصرية متعددة:
     Neon Command
     Glass Studio
     Tactical Board
     Split Broadcast
     Clean Sports TV

8. 4K/OBS Mode
   - إعدادات خاصة للبث:
     scale عام
     safe margins
     تقليل blur إذا أثر على الأداء
     تعطيل الأنميشن الثقيل

9. Data Confidence Badge
   - شارة صغيرة تقول:
     Live
     Cached
     Final
     Last update 60s

10. Match Archive Replay
   - بعد نهاية المباراة يمكن فتح نفس القالب على ملف مؤرشف من GitHub وإعادة تشغيل تطور المباراة كتحليل بعدي.

## 13. أفضل مسار تطوير قادم

الأولوية المقترحة:

1. تفعيل GitHub archive بالتوكن.
2. دمج Pull Request إلى `main`.
3. إضافة كاش صور لباقي فرق La Liga.
4. إضافة اختيار الفريق/الدوري في إعدادات القالب.
5. إضافة نظام أرشيف موسمي داخل الموقع لقراءة ملفات `match-archive`.
6. بناء قالب `Match Replay Analysis` من نفس البيانات المؤرشفة.
7. تطوير `Story Engine` حتى تتحول الأرقام إلى سرد بصري مفهوم للمشاهد.

## 14. ملاحظات أمان وتشغيل

- لا تخزن توكن GitHub في المتصفح.
- لا تضع التوكن داخل Vercel env إذا كانت الأرشفة تتم من Google Cloud.
- لا ترفع ملفات `.env` إلى GitHub.
- لا تجعل الاستخراج عدوانيا أو سريع جدا حتى لا يسبب حظرا.
- الأفضل استخدام كاش، معدل تحديث منطقي، ومصادر رسمية أو مرخصة عندما تصبح المنظومة تجارية.
