## الجزء الأول: قاعدة الصوت الثابتة للمطورين 🔊

### القاعدة المُطبّقة الآن (TypeScript Level)

```
╔══════════════════════════════════════════════════════════════════╗
║  ⚠️  AUDIO ARCHITECTURE RULE                                     ║
║                                                                   ║
║  ENTRY / EXIT  → فقط من OverlayRenderer.tsx                       ║
║  TRANSITION    → مسموح من أي Renderer داخلي                       ║
║  TTS / Custom  → مسموح بعد تأخير 700ms                            ║
║                                                                   ║
║  ❌ ممنوع: playSound('ENTRY') أو playSound('EXIT') من renderer     ║
║  ✅ مسموح: playSound('TRANSITION') فقط                             ║
╚══════════════════════════════════════════════════════════════════╝
```

### التنفيذ التقني

**RendererProps.playSound** تم تقييده على مستوى TypeScript:

```typescript
// قبل (يسمح بـ ENTRY/EXIT — خطير):
playSound: (type: 'ENTRY' | 'TRANSITION' | 'EXIT') => Promise<void>;

// بعد (يسمح فقط بـ TRANSITION — آمن):
playSound: (type: 'TRANSITION') => Promise<void>;
```

هذا يعني: **أي مطور يحاول كتابة `playSound('ENTRY')` سيحصل على خطأ TypeScript فوراً**.

## الجزء الثاني: حماية React Strict Mode 🛡️

### المشكلة
React Strict Mode في التطوير يعيد تنفيذ `useEffect` مرتين. بدون حماية:
- ENTRY يُشغّل مرتين
- EXIT يُشغّل مرتين
- أصوات مكررة حتى بعد حذف التكرار من renderers

### الحماية المُطبّقة (3 طبقات)

#### 1. `previousVisibilityRef` — حالة سابقة حقيقية
```typescript
const previousVisibilityRef = useRef<boolean | null>(null);
// يتحقق أن الانتقال فعلي (false→true أو true→false)
// يتجاهل إعادة التنفيذ (true→true) من Strict Mode
```

#### 2. `lastSoundAt` — debounce 300ms
```typescript
const lastSoundAt = useRef(0);
const SOUND_DEBOUNCE_MS = 300;
// يمنع تشغيل نفس الصوت إذا مر أقل من 300ms
// يحمي من أوامر متتالية سريعة (Stream Deck spam)
```

#### 3. `lastCommandId` — حماية unmount
```typescript
const lastCommandId = useRef(0);
const commandId = ++lastCommandId.current;
// يضمن أن setTimeout لا يطبق unmount من أمر قديم
// يحمي من race condition عند TAKE IN → TAKE OUT بسرعة
```

### تسلسل الحماية

```
config.isVisible يتغير
    ↓
هل prevVisible !== true? (Strict Mode guard)
    ↓ نعم
هل مر > 300ms من آخر صوت? (debounce guard)
    ↓ نعم
شغّل ENTRY + حفظ timestamp
    ↓
إذا EXIT: setTimeout مع commandId guard
```

## الجزء الثالث: تأخير TTS 🎤

### المشكلة
ExclusiveAlert كان يشغل TTS بعد 400ms. هذا يتداخل مع صوت ENTRY.

### الحل
```
ENTRY sound (من OverlayRenderer)
    ↓ ~500ms (مدة synth cue)
↓ 200ms فراغ آمن
TTS يبدأ عند 700ms
```

تم تغيير `setTimeout(..., 400)` → `setTimeout(..., 700)`

### fetch_fbref.py — ماذا يفعل؟

```
1. يقرأ config.json لمعرفة الدوريات المطلوبة (5 دوريات)
2. يأخذ lock file لمنع تشغيل نسختين
3. لكل دوري:
   a. requests.get(url) مع User-Agent عشوائي
   b. pd.read_html() لقراءة جداول FBref
   c. تنظيف الأعمدة + تحويل أسماء → normalized keys
   d. تحويل الأرقام + حذف التكرارات
   e. حفظ JSON في /opt/reo-data-cache/fbref/{league}-{season}.json
4. ينتظر 5 ثوانٍ بين كل طلب (rate limiting)
5. يكتب last_updated.json بملخص العملية
6. يحرر lock file
7. يكتب log كامل
```

### الحماية من عدم وجود بيانات

```
إذا الـ cache فارغ أو اللاعب غير موجود:

{
  "status": "unavailable",
  "warning": "Player not found in cache",
  "data": null
}

→ القالب يعرض "بيانات غير متوفرة" بشكل أنيق
→ لا يخترع أرقام
→ لا يكسر البث
```

### كيف نمنع scraping أثناء البث

| الآلية | الوصف |
|--------|-------|
| **Cron timing** | يعمل 06:00 و 12:00 UTC فقط (بعيد عن أوقات البث) |
| **Manual trigger** | `ssh vps && /opt/reo-datafootball-worker/run.sh` |
| **Lock file** | worker يكتب `/opt/reo-data-cache/.lock` أثناء التحديث |
| **Stale lock cleanup** | إذا Lock عمره > 30 دقيقة يُحذف تلقائياً |
| **Bridge reads only** | bridge لا يشغل أي scraping — يقرأ JSON فقط |
| **No Selenium** | FBref يعمل بـ requests فقط (لا Chrome headless) |
| **Rate limiting** | 5 ثوانٍ بين كل طلب لمنع 403 |
| **Retry + backoff** | عند 403 ينتظر 60 ثانية، عند 429 ينتظر 120 ثانية |
| **Staleness warning** | إذا الـ cache عمره > 24 ساعة يُرجع تحذير |

## الجزء السابع: لماذا ليس Cloud Function؟ ☁️❌

| الخيار | المشكلة |
|--------|---------|
| Cloud Function | لا تدعم Selenium جيداً، timeout قصير، ذاكرة محدودة |
| Cloud Run Job | ممكن لكن يحتاج Docker + تكلفة إضافية |
| **VPS Worker (المُختار)** | **مجاني (VPS موجود)، سهل، مرن، يعمل مع cron** |

المنطق:
- VPS موجود أصلاً يشغل OpenClaw + match-bridge + player-stats-bridge
- لا حاجة لـ Docker أو Cloud billing
- worker بسيط بـ Python + cron
- لا Selenium الآن (requests فقط لـ FBref)
- إذا احتجنا Selenium لاحقاً (WhoScored) → نضيفه على نفس VPS
