# تقرير مرحلة: روابط قوالب خاصة بالمستخدم عند الإنشاء

التاريخ: 2026-05-29  
اسم المرحلة: USER-SCOPED-TEMPLATE-OUTPUT-LINKS-001  
نوع المرحلة: صيانة رابط/ملكية صغيرة وآمنة  
الملفات المعدلة:

- `App.tsx`
- `utils/templateRegistry.ts`
- `types.ts`
- `pages/Library.tsx`

## 1. الهدف

كان المطلوب أن يكون لكل قالب رابط خاص به عندما يتم إنشاء القالب، وأن يبقى هذا الرابط مرتبطا بالمستخدم/الاستوديو نفسه حتى يستطيع فتحه بسرعة والتعديل أو استخدامه في OBS.

قبل هذه المرحلة كان كل قالب جديد يحصل على معرف بالشكل:

```text
instance-{Date.now()}
```

هذا يعمل تقنيا، لكنه ضعيف من ناحية المعنى:

- لا يوضح لأي استوديو/مستخدم ينتمي القالب.
- يعتمد على الوقت فقط.
- الرابط الناتج ليس مميزًا بما يكفي عند وجود أكثر من مستخدم أو أكثر من بيئة.

## 2. التشخيص

تم فحص المسار الحالي:

- إنشاء القالب يتم في `App.tsx` عبر `createOverlayFromTemplate(templateId, overlays)`.
- `createOverlayFromTemplate` موجود في `utils/templateRegistry.ts`.
- نسخ رابط القالب يتم من `pages/Library.tsx` عبر `syncManager.prepareOutputUrl(overlay.id, overlay)`.
- رابط OBS الفردي يستخدم:

```text
#/output/{overlay.id}
```

الاستنتاج:

لا نحتاج endpoint جديد ولا تغيير مسار العرض. يكفي جعل `overlay.id` للقوالب الجديدة أكثر ثباتا وارتباطا بالاستوديو.

## 3. ما تم تنفيذه

### 3.1 تمرير معرف الاستوديو عند إنشاء القالب

في `App.tsx` أصبح إنشاء القالب يمرر `syncManager.getStudioId()` إلى `createOverlayFromTemplate`.

هذا يعني أن القوالب الجديدة تعرف الاستوديو الذي أنشأها منذ لحظة الإنشاء.

### 3.2 توليد ID خاص بالمستخدم/الاستوديو

في `utils/templateRegistry.ts` تغير توليد معرف القالب الجديد من:

```text
instance-{Date.now()}
```

إلى صيغة:

```text
instance-{studioSlug}-{templateSlug}-{timeBase36}-{randomSuffix}
```

مثال تقريبي:

```text
instance-studio-abc-template-mercato-x-ray-lx9k2a-3f91c0aa
```

النتيجة:

- الرابط يبقى ثابتًا لنفس نسخة القالب بعد إنشائها.
- كل نسخة قالب جديدة تحصل على ID مستقل.
- ID يحمل بصمة الاستوديو بدون تغيير templateId.
- احتمالية التصادم أقل بكثير بسبب suffix عشوائي.

### 3.3 Web Crypto للجزء العشوائي

تم استخدام `crypto.getRandomValues` عند توفره لتوليد suffix أقوى.

يوجد fallback إلى `Math.random` فقط إذا لم تكن Web Crypto متاحة.

### 3.4 إضافة metadata اختيارية

في `types.ts` تمت إضافة:

```ts
ownerStudioId?: string;
```

هذا لا يغير الرندر ولا يفرض صلاحيات. هو metadata تساعد في تتبع ملكية النسخة الجديدة.

### 3.5 تحسين نص زر الرابط في المكتبة

في `pages/Library.tsx` تم تغيير نص زر نسخ الرابط من:

```text
OBS URL
```

إلى:

```text
رابط القالب
```

والـ tooltip أصبح:

```text
نسخ رابط القالب الخاص للمستخدم في OBS
```

هذا يجعل الواجهة أقرب لفكرة: لكل قالب رابط خاص به.

## 4. ما لم يتم تغييره

لم يتم تغيير:

- أي `templateId`.
- أي `OverlayType`.
- أي renderer.
- `components/renderers`.
- `services/audioEngine.ts`.
- `api`.
- Stream Deck.
- Player Intel V2 logic.
- رابط البرنامج العام `Program OBS`.
- روابط القوالب القديمة الموجودة مسبقا.

## 5. أثر التغيير على الروابط القديمة

القوالب القديمة التي تم إنشاؤها سابقا ومعرفاتها مثل:

```text
instance-177...
```

لن تتغير. روابطها القديمة تبقى كما هي.

التغيير يطبق فقط على القوالب الجديدة التي يتم إنشاؤها بعد هذه المرحلة.

## 6. ملاحظة مهمة عن الأمان

هذه المرحلة لا تضيف نظام صلاحيات جديدا ولا تجعل الرابط سرا مشفرا.

هي تجعل معرف القالب:

- مملوكًا عمليًا للاستوديو.
- أقل عرضة للتصادم.
- أوضح في التتبع والدعم.

إذا كان المطلوب لاحقا حماية صارمة لكل رابط، فهذا يحتاج مرحلة منفصلة تشمل token viewer أو access key لكل overlay، مع تدقيق OBS وStream Deck وVercel functions.

## 9. QA يدوي مقترح

بعد فتح الواجهة بترخيص صالح:

1. أنشئ قالبا جديدا من المكتبة.
2. افتح القالب في “قوالبي”.
3. اضغط `رابط القالب`.
4. تأكد أن الرابط يحتوي `#/output/instance-...`.
5. تأكد أن بداية الـ ID تحتوي slug للاستوديو وslug للقالب.
6. ألصق الرابط في متصفح أو OBS.
7. عدّل القالب وتأكد أن نفس الرابط يستقبل التحديثات.
8. أنشئ نسخة ثانية من نفس القالب وتأكد أنها تحصل على رابط مختلف.
9. تأكد أن `Program OBS` ما زال يعمل كرابط عام واحد لكل القوالب الحية.

## 10. التوصية التالية

بعد هذه المرحلة، أفضل خطوة قادمة هي:

1. تحسين صفحة/تجربة “قوالبي” لعرض ID أو حالة الرابط بشكل أوضح.
2. ثم مرحلة منفصلة لتثبيت viewer token لكل overlay إذا أردنا حماية أقوى.
3. ثم العودة إلى تحسينات Mercato visual polish قالبا قالبا.

لا أنصح بخلط حماية الروابط مع تحسينات الميركاتو في commit واحد.
