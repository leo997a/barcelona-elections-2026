# تقرير مرحلة: روابط تعديل ثابتة لكل قالب مستخدم

**التاريخ:** 2026-05-28  
**اسم المرحلة:** TEMPLATE-EDIT-DEEP-LINKS-001  
**نوع المرحلة:** UX Hotfix / Navigation  
**النطاق:** إضافة رابط تعديل ثابت لكل قالب محفوظ لدى المستخدم بدون endpoint جديد وبدون تغيير template IDs.

---

## 1. سبب المرحلة

طلب المستخدم أن يكون لكل قالب رابط خاص عند إنشائه، بحيث يبقى الرابط لهذا المستخدم ويمكن فتحه بسرعة للرجوع إلى القالب وتعديله.

قبل هذه المرحلة:

- فتح القالب للتعديل كان يعتمد على state داخلي فقط: `selectedOverlayId`.
- إذا أراد المستخدم الرجوع لقالب محدد لاحقًا، لا يوجد رابط مباشر يفتحه داخل الـ Editor.
- روابط OBS كانت موجودة، لكنها روابط عرض بث وليست روابط تعديل.

المطلوب كان رابط تعديل وليس رابط بث.

---

## 2. التشخيص

تم فحص المسار الحالي:

- `App.tsx` يفتح الـ Editor عندما تكون `selectedOverlayId` موجودة.
- `Library.tsx` يستدعي `onSelect(overlay.id)` عند اختيار قالب.
- `syncManager.ts` يحفظ القوالب محليًا في `localStorage` عبر `rge_overlays`.
- `OverlayConfig.id` الحالي ثابت بعد إنشاء القالب، مثل `instance-...`.

الاستنتاج:

يمكن بناء رابط آمن من نوع:

```text
/Library#/edit/{overlayId}
```

هذا الرابط لا ينشر بيانات القالب، ولا يحتاج endpoint، ويعمل للمستخدم نفسه لأن القالب موجود في تخزينه المحلي.

---

## 3. ما تم تنفيذه

### 3.1 مسار تعديل عميق داخل التطبيق

في `App.tsx` تمت إضافة دعم:

```text
#/edit/{overlayId}
```

عند فتح الرابط:

- يبحث التطبيق عن القالب داخل `overlays`.
- إذا كان موجودًا، يفتح الـ Editor مباشرة.
- إذا لم يكن موجودًا، يبقى المستخدم داخل Library بدل كسر التطبيق.

### 3.2 تحديث URL عند فتح القالب

عند الضغط على قالب من Library أو عند إنشاء قالب جديد، يصبح عنوان المتصفح مباشرة:

```text
/Library#/edit/{overlayId}
```

بهذا يصبح الرابط ظاهرًا وقابلًا للحفظ أو النسخ.

### 3.3 تنظيف الرجوع من الـ Editor

عند الضغط على زر الرجوع داخل Editor:

- يتم إغلاق الـ Editor.
- يتم تنظيف hash التعديل.
- يعود المستخدم إلى `/Library`.

هذا يمنع مشكلة إعادة فتح نفس القالب مباشرة بسبب بقاء `#/edit/{id}` في الرابط.

### 3.4 زر نسخ رابط التعديل

في `pages/Library.tsx` تمت إضافة زر أيقونة رابط داخل بطاقة كل قالب من قوالب المستخدم.

الزر ينسخ رابط التعديل الخاص بالقالب:

```text
/Library#/edit/{overlayId}
```

مع feedback سريع داخل الزر.

### 3.5 بناء الرابط من syncManager

في `services/syncManager.ts` تمت إضافة:

```ts
buildEditUrl(overlayId)
```

الدالة تنظف query الخاص بـ OBS مثل:

- `obs`
- `rgev`
- `sync`
- `d`

حتى لا يتحول رابط التعديل إلى رابط Output بالخطأ.

## 7. القيود المهمة

هذا الرابط خاص بتخزين المستخدم الحالي:

- إذا فتحه نفس المستخدم/نفس المتصفح أو نفس بيئة الاستوديو التي تحتوي القالب، سيفتح القالب بسرعة.
- إذا فتحه مستخدم آخر لا يملك هذا القالب في التخزين المحلي، لن يرى القالب.

هذا مقصود في هذه المرحلة لأن المطلوب كان رابطًا سريعًا للمستخدم نفسه، وليس مشاركة عامة أو نظام حسابات/سيرفر.

---

## 8. المخاطر المتبقية

1. **رابط التعديل ليس مشاركة سحابية**
   - لا ينقل القالب إلى مستخدم آخر.
   - إذا أردنا مشاركة حقيقية بين مستخدمين، نحتاج تصميم secure share/export phase منفصلة.

2. **لا يوجد تنبيه مخصص إذا كان القالب غير موجود**
   - التطبيق يعود إلى Library بصمت.
   - يمكن لاحقًا إضافة رسالة: "القالب غير موجود في هذا الاستوديو".

3. **اعتماد الرابط على overlay id**
   - طالما لم يتم حذف القالب، يبقى الرابط ثابتًا.
   - إذا حذف المستخدم القالب، يصبح الرابط غير قابل للفتح.

---

## 9. التوصية التالية

الأولوية القادمة المقترحة:

1. إضافة شريط صغير داخل Editor يعرض رابط التعديل وزر Copy واضح.
2. إضافة رسالة لطيفة عند فتح رابط لقالب غير موجود.
3. بعدها العودة إلى مرحلة Operator/Program Output policy: هل يسمح بأكثر من قالب LIVE أم يتم إخراج السابق تلقائيًا.

لا أنصح الآن ببناء نظام مشاركة سحابي قبل تثبيت تجربة الروابط المحلية، لأن ذلك سيحتاج صلاحيات وتخزين وقيود أمان منفصلة.
